# Container-Orchestrierung

## Warum braucht man Container-Orchestrierung?

- Viele Container müssen gleichzeitig verwaltet werden
- Automatisches Starten, Stoppen und Neustarten von Containern
- Verteilung der Container auf mehrere Server
- Bessere Ausfallsicherheit
- Einfache Skalierung bei mehr Last
- Automatische Updates und Rollbacks
- Effizientere Nutzung von Ressourcen
- Weniger manueller Verwaltungsaufwand

## Wie funktioniert Container-Orchestrierung?

- Container werden in einer Umgebung mit mehreren Servern verwaltet
- Ein Orchestrator entscheidet, wo Container laufen sollen
- Anwendungen werden meist über Konfigurationsdateien beschrieben
- Der gewünschte Zustand wird definiert, zum Beispiel: „3 Instanzen sollen laufen“
- Der Orchestrator überwacht laufend den tatsächlichen Zustand
- Bei Fehlern werden Container automatisch neu gestartet oder ersetzt
- Netzwerk, Speicher, Skalierung und Updates werden zentral gesteuert

## Was versteht man unter horizontaler Skalierung im Kontext von Cloud-Anwendungen?

- Mehrere Instanzen einer Anwendung werden parallel gestartet
- Last wird auf mehrere Container, Pods oder Server verteilt
- Beispiel: Statt einem Webserver laufen fünf Webserver gleichzeitig
- Neue Instanzen können bei hoher Last automatisch hinzugefügt werden
- Bei geringer Last können Instanzen wieder entfernt werden
- Ziel: bessere Performance, Verfügbarkeit und Ausfallsicherheit

## Was gibt es für Deployment-Strategien?

- **Rolling Deployment:** Schrittweise Aktualisierung einzelner Instanzen
- **Blue-Green Deployment:** Zwei Umgebungen, eine aktiv und eine vorbereitet
- **Canary Deployment:** Neue Version zuerst nur für wenige Nutzer
- **Recreate Deployment:** Alte Version wird beendet, danach neue gestartet
- **A/B Testing:** Unterschiedliche Versionen für verschiedene Nutzergruppen
- **Shadow Deployment:** Neue Version läuft parallel, verarbeitet aber noch keinen echten Nutzerverkehr direkt
- **Feature Toggles:** Funktionen werden per Schalter aktiviert oder deaktiviert
